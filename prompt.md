# Datamancy Stack: Operator Notes

## Overview

- This repository is a **Docker Compose stack generator**.
- The build script is `build-datamancy-v3.kts`.
- Build outputs are written to `dist/`.
- The final generated Compose file is `dist/docker-compose.yml`.
- `smart-up.sh` is included in `dist/` and should be used instead of normal `docker compose up`, because it only starts changed containers and is much faster.

## Repository structure

### Core stack templates

- `global.settings/` — templates for global stack settings and shared configuration.
- `stack.compose/` — templates for individual stack services.
- `stack.config/` — templates for configuration files mounted or copied into containers.
- `stack.containers/` — Dockerfile sources for custom container images.
- `stack.kotlin/` — Kotlin source for custom services.

### Test-related templates and sources

- `tests.compose/` — Compose templates for test-only services.
- `tests.config/` — configuration templates used by tests.
- `tests.containers/` — sources for test containers, including the test runner.
- `tests.kotlin/` — Kotlin test sources, including test-runner-related code.

- A test runner exists at:
    - `tests.containers/test-runner`
- The help script for the test runner on the server is:
    - `~/datamancy/run-tests.sh`

## Operating rules

- Run **all tests**.
- Do **not** update the server before testing; it has already been updated.
- Analyze the screenshots generated by the test runner.
- Diagnose issues experimentally by testing directly on the server.
- Take initiative in fixing problems.
- Be humble in changes and conclusions.
- If waiting is necessary, wait in **120-second intervals**.
- Do **not** use `-v` with `rsync`.
- Keep the Git working tree under control:
    - either commit changes, or
    - update `.gitignore` appropriately,
    - because the build script uses Git for versioning.
- Do not write long reports.

## Stack introspection and self-awareness

### Source and workspace

- The stack should bundle its own source code in a **Forgejo** repository.
- The stack should provide a containerized workspace through:
    - a virtualized Docker socket
    - an LLM server

### MCP / agent integration

- `agent-tool-server` should be:
    - renamed to **model-context-server**
    - rewired to conform to the **Model Context Protocol**
- Any language model using the MCP server should be able to:
    - gather stack context
    - introspect the stack
    - call stack tools

### Authentication and authorization

- MCP authentication should be tied to LDAP accounts.
- Each agent should have its own LDAP account.
- LDAP roles should determine what the agent can access.

## Observability and integrations

- Internal systems should support modular integration into Grafana.
- Home Assistant is one example that should be considered for easy integration.
- Review which services should be exposed in Grafana and configure them appropriately.

## Identity and access management

- Account creation, removal, and role control should be centralized.
- These flows need thorough testing.
